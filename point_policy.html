적립 정책

2-1적립 발생 조건 및 시점
  • KAYAK 아웃링크를 통해 타 OTA에서 예약 완료
  • 적립 시점: 체크아웃일 + 7일 (자사 배치 처리) — KAYAK 정산 완료 여부와 무관하게 자사가 먼저 지급
  • 배치 주기: 일 1회 (매일 자정, 체크아웃일 기준 +7일 도달한 건 일괄 적립)
배경: KAYAK 정산 시점까지 기다리면 고객 불만이 쌓일 수 있으므로 자사 기준으로 선적립. 이후 KAYAK Cancelled 전환 시 회수 처리(3-3절 참조).

⚠️ 해외 접속 유저 예외 (v0.2 추가)

  • 한국 유저라도 해외에서 접속하면 KAYAK이 해당 예약의 ClientCountryCode를 KR이 아닌 접속 국가로 판단
  • 결과: 포인트 미발생 — 자사 소급 적립 불가
  • CS 안내: "해외 접속 환경에서는 포인트가 적립되지 않을 수 있습니다"

🔧 x-original-client-ip 헤더 필수 전달 (v0.2 추가)

  • Hotels Search API 호출 시 x-original-client-ip 헤더에 유저의 실제 접속 IP 반드시 전달
  • 미전달 시 KAYAK이 서버 IP를 Client Country로 오인 → 캐시백 미발생 가능
  • 개발팀 구현 필수 항목 (7절 미결 #6 참조)

3절

감소 정책

MVP에서 발생하는 감소 케이스는 아래 4가지.

이벤트 타입발생 조건처리
booking_cancel 선적립 이전에 KAYAK Cancelled 전환 (체크아웃+7일 미도래) 적립 없이 카드 상태만 취소·만료로 변경
booking_clawback 선적립 이후 KAYAK Cancelled 전환 (체크아웃+7일 이후) 선적립 포인트 회수 (차감)
expiry 적립일 기준 +1년 경과 소멸 대상 포인트 자동 처리 (배치)
withdrawal 회원 탈퇴 처리 시 잔여 포인트 전액 소멸
3-1취소 케이스 분기
취소 시점처리 방식
체크아웃+7일 이전 취소 (선적립 전) KAYAK Cancelled 감지 → 적립 없이 카드 상태만 취소·만료 표시. booking_cancel
체크아웃+7일 이후 취소 (선적립 후) KAYAK Cancelled 감지 → 선적립 포인트 회수. booking_clawback
MVP에서는 포인트 사용 기능이 없으므로 회수 시 잔액 부족 케이스 발생하지 않음. 회수 로직 단순 구현 가능.
3-2기간 소멸 배치
  • 기준: 각 적립 건의 적립일(booking_earn 처리일) + 1년
  • 처리 주기: 일 1회 (매일 자정 소멸 대상 일괄 처리)
  • 처리 내용: 소멸 대상 건을 이벤트 타입 expiry로 내역 기록 후 잔액 차감
3-3선적립 회수 배치 (booking_clawback)
  • KAYAK Reporting API를 주기적으로 조회하여 Cancelled 전환 건 감지
  • 자사 이벤트 타입 booking_clawback으로 차감 내역 기록
  • 회수 후 MY02 카드 배지: 취소·만료
3-4탈퇴 소멸
  • 회원 탈퇴 요청 처리 시 잔여 포인트 전액 이벤트 타입 withdrawal로 소멸 처리
  • 탈퇴 완료 후 복구 불가

#1

#4

#5
해외 접속 유저 포인트 배지 표시 정책
한국 유저가 해외에서 접속 시 포인트 배지(💸)를 숙소상세(S02)에서 어떻게 표시할지 — 배지 숨김 vs 그대로 표시 후 적립 안내 문구 추가. 개발팀 실측 확인 필요.
#6
x-original-client-ip 헤더 전달 방식
Hotels Search API 호출 시 유저 실제 IP를 x-original-client-ip 헤더로 전달하는 구현 방식 확인 — 프록시·CDN 환경에서 실제 IP 추출 방법. 개발팀 확인 필요.
