제1조

개요

1-1Phase 2 목표

Phase 1(KAYAK API 기반 메타서치) 완료 후, 올마이투어 자체 공급사 연동을 통해 가격 비교 목록에 자사 상품을 포함시킨다.

Phase 2 주요 연동 대상: 올마이투어 써드파티 공급사 (익스피디아 우선)

1-2올마이투어 상품 구분
구분대상처리 방식Phase
써드파티 익스피디아, 호텔베즈, DIDA 트래블 등 API 호출 Phase 2 부터 순차 연결
직계약 올마이투어 직계약 호텔 DB 쿼리 별도 논의 필요
제2조

매핑 구조

2-1기본 매핑 테이블 구조
KAYAK 호텔 ID ↔ 올마이투어 호텔 ID ↔ 공급사(익스피디아 등) 호텔 ID
  • KAYAK 호텔 ID와 공급사 호텔 ID를 직접 매핑하지 않음
  • 올마이투어 내부 ID를 허브로 하여 공급사 추가 시 확장 가능한 구조 유지
2-2매핑 범위
  • Phase 2 매핑 대상: 호텔 매핑만
  • city코드 매핑 불필요 — 검색결과 숙소 목록은 KAYAK API가 내려주므로, KAYAK city코드를 올마이투어 도시 코드에 별도 매핑하지 않아도 됨
  • city코드 매핑은 Phase 3 이상(KAYAK 미보유 숙소 서비스 시)에서 검토
2-3매핑 데이터 출처
  • KAYAK Static Data Feed API로 전체 호텔 데이터 적재 (호텔 코드, city코드 포함)
  • 예상 매핑률: 약 60% 수준 (스테이피아 과거 경험 기반)
  • 매핑 미완료 숙소: 공급사 병렬 호출 없이 KAYAK 결과만 노출
2-4매핑 작업
  • VerbTech 또는 AI 기반 자동 매핑 도구 활용 검토
  • 담당: 차도현 연구원 (진행 중)
제3조

검색 플로우

3-1Autocomplete
  • Autocomplete API: KAYAK API 그대로 사용
  • placeType 11개 모두 노출 (별도 필터링 없음)
  • Autocomplete를 위한 별도 city코드 매핑 불필요
3-2Autocomplete 선택 → 검색결과 페이지(S01) 이동
  • 도시(city코드) 선택 → S01 (해당 도시 숙소 목록 표시)
  • 호텔(hotel) 선택 → S01 (선택한 호텔 최상단, 이하 동일 도시 기준 목록)
3-3병렬 호출 구조
1. Autocomplete 키워드 입력 → KAYAK Autocomplete API 호출 2. 도시 또는 호텔 선택 → S01으로 이동 3. [병렬 호출] A. KAYAK 검색 API → 숙소 목록 50개 수령 B. 50개 호텔 ID → 올마이투어 매핑 테이블 조회 → 공급사 호텔 코드 획득 C. 공급사 API 호출 → 가격 수령 4. KAYAK 응답 포맷에 공급사 Provider 인덱스 추가하여 병합 5. 가격 기준 정렬 → 프론트 전달
병렬 호출 흐름도 — 시각적 다이어그램은 kayak_allmytour_flow.html을 참조하세요.
3-4페이지당 숙소 수
  • 50개 유지
제4조

써드파티 연결 정책

4-1연결 우선순위
순서공급사상태
1 익스피디아 Phase 2 최우선
2 호텔베즈 Phase 2 이후 순차 (논의필요)
3 DIDA 트래블 Phase 2 이후 순차 (논의필요)
4-2Provider 인덱스 구성
  • KAYAK 응답 포맷의 Provider 인덱스에 올마이투어 공급사 코드를 별도 추가
  • 각 공급사별 인덱스로 클릭 시 연결 대상(자사 상세 페이지 / 아웃링크) 구분
4-3Rate Limit 사전 확인 필수
⚠ 연동 전 반드시 확인: 올마이투어 써드파티 공급사 연결 전 각 공급사의 Rate Limit 정책을 사전 확인해야 합니다.
  • 확인 항목: N개 숙소 일괄 조회 가능 여부 vs 1건씩 N회 호출 필요 여부
  • 대응: 공급사별 조건 상이 → API 문서·계약 조건 사전 파악 후 연동 일정 수립
제5조

화면·어드민 기획 방향

5-1추가 필요 화면

익스피디아 상품은 올마이투어 자사 OTA 상품으로 처리하므로 결제 라인 자체 구성이 필요하다.

화면내용
결제 라인 포트원 기반 자체 결제 구성
예약 내역 예정 / 완료 / 취소
어드민 결제 관리 공급사별 예약·정산 관리
5-2자사 상품 UI 표시 방식
  • Vio.com 참고: 숙소 상세 내 자사 상품 섹션 별도 표시
  • 검색결과 목록 카드 수에는 영향 없음 (50개 유지)
  • 자사 상품 최상단 고정 방식 채택하지 않음 (가격 기준 정렬 유지)
미결: 자사 상품 UI 표시 방식 확정 필요 (별도 섹션 vs 리스트 내 정렬). 미결 이슈 #2 참조.

7-2Expedia Lodging 지원 언어 범위

Expedia Lodging API는 전체 언어를 지원하지 않음. 1차 지원 UI 언어 7종은 모두 Lodging 지원 범위 내.

언어Expedia 코드 → API 코드Lodging 지원
한국어ko-KR → ko_KR✅
일본어ja-JP → ja_JP✅
영어en-US → en_US / en-GB → en_GB✅
스페인어(멕시코)es-MX → es_MX✅
인도네시아어id-ID → id_ID✅
베트남어vi-VN → vi_VN✅
중국어 번체(대만)zh-TW → zh_TW✅

폴백 규칙: 7종 범위 외 언어로 Expedia API 호출 시 → en_US로 폴백 처리

중국어 코드 주의: zh-CN(간체/본토)·zh-TW(번체/대만)·zh-HK(광둥어/홍콩) 별개 코드. 혼용 금지. zh-CN은 중국 본토 ICP 라이선스 요건으로 서비스 불가, 제외 유지.
